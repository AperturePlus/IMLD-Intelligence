## 产品技术方案

本章以“评委老师可快速理解”为目标，从业务价值、技术落地和合规可行性三个角度说明系统方案。我们不把技术堆砌成术语清单，而是重点回答三个问题：为什么这样设计、这样设计能解决什么现实问题、以及这套方案在医院场景里是否真的可运行。

### 系统架构概述

项目采用模块化单体架构，核心思路是“同一套代码同时服务 SaaS 与私有化医院”。这一选择不是为了追求新概念，而是为了平衡两类客户的真实差异：基层机构更关注上线速度和成本，大型医院更关注数据主权和合规边界。如果维护两套系统，研发和运维成本会在后期迅速失控；因此我们从一开始就把“单代码基线”作为工程原则，通过配置切换运行形态。

#### 前端架构

本商业计划书当前版本聚焦后端与数据库，前端界面体验和视觉设计由前端团队单独提交。为保证协作效率，后端已经提前提供接口契约、字段字典、错误码规范和环境配置规则，因此前端可以在不等待后端反复解释的情况下完成开发与联调。

#### 后端架构

后端以 11 个业务模块组织能力，分别覆盖身份权限、临床数据、诊断引擎、随访管理、筛查转介、报告管理、会员支付、通知、系统集成、审计追踪和许可证管理。模块化结构带来的直接收益是：每个模块可独立演进，团队分工清晰，且不会因为某一块需求变动牵连全局。更重要的是，这种架构天然适合医疗行业“高合规、长周期”的特点，后续做功能审计、责任追溯和版本回滚都更可控。

#### 系统部署

系统在部署层面提供三种模式：`saas` 适合基层机构快速接入，`private` 适合大型医院内网独立运行，`hybrid` 用于在院内独立运行基础上做受控云协同。三种模式都来自同一代码仓，不存在“私有化是另一套代码”的风险。在私有化模式下，核心诊疗流程可以离线完成；在混合模式下，只有白名单场景允许出域，且通信策略可强制启用 mTLS、请求签名与防重放机制，保证“可协同但不失控”。

### 核心算法介绍

系统并不把算法做成一个“黑盒按钮”，而是采用可插拔诊断编排机制。具体来说，平台先通过模型注册中心管理不同类型模型（规则、机器学习、NLP、LLM），再通过诊断会话串联输入快照、候选结果、推荐建议与医生反馈。这样做的价值在于：即使后续更换模型供应商，业务流程和数据库结构也不需要推倒重来。

从评委视角看，这套算法方案最大的特点是“可解释、可追溯、可持续优化”。每次诊断都能知道使用了哪个模型版本、为什么给出这个建议、医生最终是否采纳；模型调用信息进入专门审计日志，记录耗时和成本，但不会存储敏感原文，兼顾了隐私安全与运营管理。也就是说，这不是单纯追求“模型分数高”，而是追求“医疗场景可落地”。

### 数据库方案

数据库不是简单的“存数据工具”，而是产品可交付性的核心。医疗项目往往失败在两个点：一是后期跨系统数据对不齐，二是合规审计时拿不出完整证据链。我们的数据库方案正是围绕这两个问题展开。

#### 设计原则

数据库设计遵循“业务真实、隔离明确、审计完整、安全优先”四个原则。表结构直接对应医疗流程，从患者建档、就诊、诊断、报告到随访都能形成闭环；多租户字段从底层约束数据边界；敏感信息优先采用加密字段设计；关键行为必须可审计、可追溯。这意味着系统在功能上线时就具备合规能力，而不是事后补洞。

#### 分域设计

当前数据库按四个业务域管理，分别是身份与临床基础、诊断与报告、随访筛查与支付、通知集成与审计，合计 46 张核心表。这样分域的好处是非常务实的：一方面便于开发协作，团队可以按域并行推进；另一方面便于后期扩展，未来无论做院内深度定制，还是按域拆分服务，都有清晰边界，不会出现“大表互相耦合、改一处动全身”的局面。

#### 存储设计

在存储策略上，系统采用“强结构字段 + 扩展 JSON 字段”组合。强结构字段用于保证核心业务一致性，扩展字段用于承载医疗场景中频繁变化的结构化信息，例如诊断输入快照、问卷评分规则、审计前后镜像。对于高敏信息，系统通过加密字段和身份映射隔离降低直接暴露风险；对于医疗文书，报告版本快照机制确保历史可复盘、责任可追溯。

#### 优化方案

性能优化并非停留在口号层面。当前方案已经落地了 20 个核心索引，覆盖患者、就诊、诊断、随访、审计等高频路径；查询侧默认结合租户条件与业务条件，避免无边界扫描；审计接口自带分页上限和大载荷摘要化机制，防止日志型数据冲垮存储。在后续扩展中，可继续按时间分区审计数据、引入缓存热点策略和异步削峰机制，从而保证系统在业务增长后仍可稳定运行。

### 开发平台

开发平台强调“可复现与可迁移”，即在学生团队、企业环境、医院交付环境之间都能稳定运行，不依赖少数人的本地经验。

#### 硬件开发平台

后端与数据库研发不依赖昂贵 GPU，普通 8 核 CPU、16GB 内存的开发机即可覆盖日常开发、联调和集成测试。对于答辩和评审，这一点非常关键：项目可以在常规教学与实践环境中稳定复现，降低了“只能演示不能交付”的风险。

#### 软件开发平台

当前平台基于 Java 21、Spring Boot 3.5.10、MyBatis 3.0.5 与 Gradle Kotlin DSL，数据库默认 PostgreSQL，同时兼容 MariaDB；缓存和消息组件使用 Redis 与 Kafka。测试侧采用 JUnit5、MockMvc 和 Testcontainers。整体技术栈成熟、稳定、社区支持充分，适合医疗项目长期维护，不会把未来运维压力转嫁给客户医院。

### 开发工具与技术架构

工具链选择遵循“稳定优先、可审计、可持续维护”的原则。我们不是追求炫技，而是确保每一项技术都能解释清楚“为什么需要它”。

#### 前端技术栈

前端技术选型由前端团队独立负责；后端在本计划书中主要承担接口契约与联调支持职责，确保前后端并行推进时不会因协议反复变更造成进度失控。

#### 后端技术栈

后端以 Spring Boot 作为基础框架，以 MyBatis 保障 SQL 可控性，以 AOP 机制降低审计埋点成本，并为同步业务接口与异步协同场景分别预留了 MVC 与 WebFlux 能力。该组合最大优势是“工程透明”：业务逻辑、SQL、配置、审计链路都可直观检查，便于后期合规审查和医院 IT 接手运维。

#### 数据库与缓存技术

数据库层选择关系型主库承载交易一致性，JSON 字段承载业务弹性；缓存用于降低热点读取压力；消息组件用于异步解耦和协同扩展。由于医疗系统对一致性要求高，我们坚持“数据库主责任、缓存辅助增效”，避免把关键一致性压在不可控缓存策略上。

#### 可靠性支持

可靠性设计强调“故障可控”。系统预置熔断、重试、限流和超时策略，避免下游故障扩散；关键流程依赖事务回滚保护数据一致性；审计采用 fail-closed 策略，保证关键操作不能在无记录状态下通过。这些机制共同构成了医疗系统必须具备的“安全失败能力”。

#### 版本控制与管理

版本治理坚持单仓主干演进，环境差异通过配置切换实现，避免长期维护“双轨代码”。私有化授权采用签名许可证，升级采用签名发布清单并结合支持期判定，从技术上保证商业规则可执行、可审计，而不是依赖人工口头约束。

#### DevOps

DevOps 流程围绕“自动化构建、自动化测试、可追溯发布”展开。通过统一构建脚本和分环境配置，团队能够把同一套代码交付到 SaaS、私有化和混合模式。对评委来说，这意味着项目不仅有设计图，也具备可重复交付的方法论。

## 产品功能实现

本章从业务闭环角度说明系统已经具备的后端能力，不以“功能点罗列”为主，而以“医疗机构实际能用什么”作为叙述主线。

### 功能总体设计说明

系统功能分为三层：第一层是核心诊疗闭环，覆盖患者、就诊、诊断、报告、随访；第二层是协同增值能力，支持筛查转介、会员服务、消息通知与跨系统对接；第三层是安全合规底座，包含权限、审计、许可证与升级管控。三层协同后，既能支撑医院日常业务，也能支撑平台商业化运营，不会出现“业务可用但合规不可过”或“合规完整但业务无法落地”的割裂问题。

### 核心业务模块实现

身份与权限模块负责“谁可以做什么”，已覆盖租户、用户、角色、ABAC 策略、患者身份映射和知情同意管理；临床模块负责“数据从哪里来、是否可用于决策”，已覆盖检验、病史、基因、影像等关键来源；诊断与报告模块负责“如何形成可解释结论并沉淀可追溯文书”，通过模型注册、诊断会话、结果建议和报告版本机制形成闭环；随访模块负责“院后管理是否可执行”，通过计划、任务、上报、告警和处置动作让慢病管理从表格化走向流程化。换句话说，核心模块已经形成从数据采集到临床行动的完整主链路。

### 辅助功能模块实现

辅助模块承担“让核心业务持续运转并具备增长能力”的职责。筛查模块支持院外问卷与院内转介衔接，帮助项目建立入口；支付与会员模块支撑增值服务闭环，帮助项目形成可持续商业模式；通知模块记录从发送到回执的全过程，确保患者触达可评估；集成模块对接第三方系统任务并保留执行状态，为后期失败补偿与运维定位提供依据；审计与许可证模块则把合规和商业授权规则技术化，保证交付后可持续运营。

### 系统交互与接口设计

系统接口设计强调“边界清晰、责任明确”。当前已落地审计查询接口，支持按租户、角色、时间窗口与分页条件进行检索，并通过请求头机制统一携带租户和角色上下文。跨域协同遵循事件契约思路，不允许通过共享数据库破坏边界；在私有化医院场景中，核心流程可独立运行，云协同只在白名单业务中开启。这种设计既保障了业务可用性，也降低了安全与合规风险。

### 安全与权限控制

安全控制是系统的底座能力。首先，多租户隔离从数据层实施，避免跨机构数据串读；其次，权限控制结合角色与属性策略，默认拒绝未授权访问；再次，审计链路完整记录操作行为、敏感访问与模型调用，并对敏感字段自动脱敏；最后，私有化模式下通过签名许可证、激活码和支持期规则控制启动与升级权限。总体上，系统不是靠“制度宣讲”保证安全，而是把安全规则直接写进系统行为。

## 产品测试计划

本章目标不是展示“测了很多用例”，而是证明系统在真实环境中能够稳定、安全、可交付地运行，尤其能够经受医疗行业对隐私与审计的高要求。

### 测试目标

测试目标围绕三个层面展开：第一，验证核心业务链路可正确执行；第二，验证安全与合规机制在异常场景下依然有效；第三，验证同一代码基线在不同部署模式下行为可控、结果一致。这三项同时成立，才意味着项目具备实际落地价值。

#### 前端测试目标

前端测试目标由前端团队单独负责。本版本后端侧的目标是提供稳定契约、可追踪日志和可复现联调环境，为前端验收提供可靠基础。

#### 后端测试目标

后端测试重点放在高风险环节：多租户隔离是否可靠、审计是否完整、许可证与升级规则是否可执行、异常是否能安全失败。通过这些测试，可以证明系统不是“演示可用”，而是“上线可控”。

### 测试范围

测试范围覆盖功能正确性、安全性、可靠性与可部署性四个维度，并与现有代码中的自动化测试形成对应关系。

#### 前端测试范围

前端页面功能、交互体验和兼容性不在本次后端交付主范围内，但后端将持续提供联调支持与接口稳定性保障。

#### 后端测试范围

后端测试覆盖 11 个业务模块的数据访问层与关键服务逻辑，涵盖 46 张核心表的读写一致性、租户边界、JSON 字段处理、约束有效性，以及审计接口、权限校验、许可证验证和升级判定等关键链路。现有测试已包含模块级集成测试、接口集成测试和核心安全逻辑单元测试。

### 测试环境

测试环境设计遵循“本地可复现、评审可验证、上线可迁移”，尽量减少环境差异带来的结论偏差。

#### 硬件环境

标准开发机即可运行主要测试流程，建议配置为 8 核 CPU、16GB 内存和足够磁盘空间；通过 Docker 支撑数据库与依赖组件启动。这说明项目不依赖稀缺硬件，具备普适部署条件。

#### 软件环境

软件环境包括 Java 21、Gradle、Spring Boot、PostgreSQL、MyBatis 与 JUnit5 生态。集成测试采用 Testcontainers 动态拉起数据库实例，避免“共享测试库污染”导致的假通过问题。

#### 网络环境

测试主要在本地或局域网环境进行，核心业务链路可在无外网条件下验证。对于混合通信能力，测试仅开放必要出站地址，严格禁止反向直连院内核心数据源，保持与实际合规要求一致。

#### 数据准备

测试数据通过标准 schema 脚本初始化，再由用例按场景构造最小数据集。这样既保证可重复执行，也能针对租户隔离、权限拒绝、审计回滚等关键场景进行精确验证，避免“样例数据看起来能跑但无法覆盖风险”。

### 前端测试设计

该部分由前端团队主导，后端侧仅说明协同机制，确保职责清楚、沟通成本可控。

#### 功能测试

后端提供稳定接口和字段语义，保证前端功能测试过程中不因接口频繁变更而反复返工。

#### 页面交互测试

后端提供可追踪请求上下文（例如 TraceId）和标准错误返回，帮助前端快速定位交互失败原因。

#### 兼容性测试

后端保证在 SaaS、私有化、混合三种模式下接口语义一致，减少前端跨环境适配成本。

#### 异常与边界测试

后端已经定义并验证关键边界行为，例如缺失租户头、角色不合法、分页超限等，为前端异常展示提供确定性依据。

### 后端测试设计

后端测试采用分层策略，强调“先守住安全底线，再验证业务完整性”。

#### 接口测试

接口测试以审计查询 API 为典型样板，重点验证租户头必填、角色白名单、分页上限和跨租户隔离行为。该类测试通过 MockMvc 在集成环境中执行，能真实反映接口层行为。

#### 业务逻辑测试

业务逻辑测试聚焦关键控制链路，包括审计自动补全与脱敏、审计失败触发事务回滚、许可证签名验证、支持期外升级拒绝与安全补丁例外策略。这些测试直接对应商业规则和合规规则是否“真正被系统执行”。

#### 数据库测试

数据库测试覆盖主数据 CRUD、租户隔离、JSON 字段读写一致性以及约束有效性。通过 Testcontainers 使用临时 PostgreSQL 实例，确保测试结果与生产行为一致，降低环境差异风险。

#### 鉴权测试

鉴权测试验证默认拒绝策略在各种输入下是否稳定生效，特别关注角色白名单、上下文缺失和敏感访问记录是否准确，确保权限体系在压力和异常下不失真。

#### 异常处理测试

异常处理测试关注系统“能否安全失败”：当审计写入失败、许可证缺失、公钥错误或参数非法时，系统是否能及时阻断并给出可定位信息，而不是悄悄放行风险操作。

### 质量属性测试

质量属性测试用于验证系统长期运行能力，确保项目不是“短期可演示”，而是“长期可运营”。

#### 性能

性能验证围绕关键查询路径展开，通过索引覆盖、分页限制和查询条件约束控制响应成本。在后续压力测试中可进一步引入 P95/P99 指标，形成可量化优化闭环。

#### 安全性

安全测试重点检查敏感信息脱敏、越权访问拦截、签名文件防篡改和出域策略有效性，确保系统在真实攻击面前具有防护能力，而不是仅在理想流程中看起来安全。

#### 可靠性

可靠性测试关注故障隔离和恢复能力，验证事务回滚、熔断重试策略和链路追踪机制在异常情况下是否稳定工作，从而保障医疗业务连续性。

### 测试验收标准

验收以“可量化、可复核”为原则：自动化测试需全部通过；多租户隔离必须无跨租户泄露；审计关键场景需完整可追溯；私有化授权与升级规则需按配置可执行；文档、配置与代码版本应保持一致并可复现实验结果。只有同时满足这些条件，才认定系统达到可交付标准。

### 测试交付产物

测试交付物包括测试计划、用例清单、执行报告、安全专项记录、缺陷闭环记录以及数据字典和字段依据说明。对挑战杯评审而言，这些材料不仅证明“我们做了测试”，更证明“我们具备工程化交付能力与风险控制能力”。

### DTO 与 HTTP 契约分层

当前仓库已经为 11 个模块补齐“契约先行”的 DTO 与 HTTP 接口设计，并统一采用 `Query / Request / Response / Shared` 分类方式（`audit` 模块保留既有响应结构，仅补查询分类）。这意味着后续控制器与前端联调将围绕稳定契约展开，而不是直接围绕数据库模型或仓储对象耦合。完整清单位于 `docs/module-dto-http-contracts.md`。

### 控制器契约与统一异常处理

在 DTO 与 HTTP 契约分层之外，当前仓库又补充了一层显式 `ControllerContract` 接口，用于约束后续控制器实现必须先承接既有协议边界，再进入业务实现阶段。与此同时，系统增加了 `GlobalExceptionHandler`，将参数校验、授权拒绝、显式状态异常和未分类运行时异常统一转换为稳定的错误响应，降低联调阶段因异常格式不一致导致的重复适配成本。
